# Cap — Feasibility Report and Phased Build Plan

This is the pre-code deliverable: an honest read on what iOS will actually let "Cap" do, and a phasing plan that builds the real version of this instead of a demo that breaks the first time you close the app.

## 1. Feasibility Table

| Capability | Real iOS constraint | Best achievable approximation |
|---|---|---|
| **Open-mic, always-listening conversation** | There is no public API for a custom wake word at the OS level — that layer is reserved for "Hey Siri." A third-party app can only listen continuously by keeping `AVAudioSession` active with a background-audio entitlement, which (a) shows the red mic indicator permanently, (b) drains battery fast, (c) invites App Store review scrutiny, and (d) still gets suspended by iOS under memory/battery pressure — there's no guarantee it survives. | Tap-to-start, not push-to-talk-per-sentence. One tap (Action Button, Lock Screen widget, Dynamic Island, AirPods press, or Siri Shortcut) opens a live session where you talk back and forth freely with no re-tapping — Cap listens until you stop talking or say "that's it." This is functionally how Siri itself works. True 24/7 ambient listening is not realistically buildable as a third-party app. |
| **Background execution / proactive reminders** | iOS does not let apps run continuously in the background. `BGTaskScheduler` is opportunistic — the OS decides if/when your task runs based on battery, usage patterns, and Low Power Mode, with no guaranteed timing. Standard background grace periods are seconds, not minutes. | Front-load the intelligence: when the app is open, it pre-computes and schedules local notifications (deadlines, leave-by alerts, briefings) for the future. Background refresh tasks opportunistically re-sync and re-schedule when the OS grants time. Geofencing and significant-location-change (system-level, not app-level background loops) wake the app reliably for location triggers, survive app termination by the system, but stop after a manual force-quit until next device unlock. Net effect: reminders fire close to on-time, not because Cap is "always running," but because it scheduled them in advance and gets occasional, OS-controlled refresh windows. |
| **Fully offline operation** | Apple's on-device Foundation Models framework (iOS 26+, Apple Intelligence devices, iPhone 15 Pro and newer) gives a real on-device ~3B-parameter LLM with tool-calling and structured output, fully offline, no data leaving the device. But it is explicitly not a general-knowledge model — it's built for language tasks (summarizing, classifying, extracting, drafting) inside your own context, not for "what's the capital of Mongolia" type recall. | Offline mode handles everything that's really "your data, reasoned over": triage, summarizing your emails/assignments, drafting replies, briefings, planning. For open-ended world-knowledge questions, Cap should say so plainly ("that one needs the cloud, I don't have signal") rather than guess with a model that isn't built for it. |

## 2. Recommended Architecture

**The brain.** Default brain is the on-device Foundation Models session — it's the only one that's free, instant, and fully private. Cloud (your choice of provider, added later) is only invoked for two narrow cases: genuine general-knowledge questions outside your own data, or tasks whose context is too large/complex for the on-device model. Every cloud call should be visibly flagged in the UI in real time ("→ cloud") — not a settings-page disclosure buried somewhere you'll never read it. Nothing routes through any third party except that single chosen cloud model on the rare occasions it's needed; there's no analytics SDK, no crash reporter that phones home, no ad framework.

**What actually leaves the device, and when.** Four categories of outbound traffic, and nothing else: (1) OAuth calls to Google/Microsoft to read your Calendar/Gmail/Outlook — this is inherent to using those services, the data goes from Google/Microsoft to your phone, not through any middleman of ours; (2) Canvas REST API calls to your school's Canvas instance, using your personal token, same logic; (3) MapKit directions requests to Apple's own routing service for leave-by calculations; (4) the occasional cloud-LLM call described above, and only the specific text needed for that one question — not your whole calendar, inbox, or memory store. Everything else — your conversation history, preferences, task list, locally-cached calendar/email/Canvas data — lives in an encrypted local store (`NSFileProtectionComplete`) on the device, with OAuth refresh tokens and the Canvas PAT in Keychain. There's a real tradeoff worth naming: because nothing syncs, there's no iCloud backup of Cap's memory either. If you lose or wipe the phone, that memory is gone, unless you manually export it (e.g., an encrypted local backup file) before that happens. That's the honest cost of the privacy principle, and worth deciding on deliberately rather than discovering by accident.

**Data flow shape.** EventKit and Canvas and (eventually) Google/Microsoft are read live or cached briefly, not duplicated into a shadow database — Cap queries the source of truth and keeps only what it needs for offline use (e.g., today's and this week's events/assignments) cached locally with a short TTL, refreshed whenever the app opens or gets a background window. The planning/triage logic, the memory of your preferences, and the persona/tone all run as local logic plus on-device model calls — there's no server component for Cap itself, which also means no "Cap backend" that could be a future privacy liability.

## 3. Phased Roadmap

**Phase 0 — Foundation (read-only, app-open only).** On-device chat via Foundation Models with the blunt/brief persona baked into the system prompt. Apple Calendar read via EventKit. Canvas integration: pull courses, assignments, due dates, and `syllabus_body` via your personal access token. Local encrypted store for tasks/notes. Quick capture via the iOS Share Sheet or a Shortcut ("send to Cap" → becomes a task). No voice, no background, no cloud yet — this phase proves the core loop works and the persona actually sounds like Cap and not a chatbot.

**Phase 1 — Voice and Planning.** Tap-to-talk session (not push-to-talk per sentence) using the Speech framework for input and `AVSpeechSynthesizer` for replies, routed to whatever audio output is active (wired/Bluetooth earbuds included for free via `AVAudioSession`). Deadline triage that weighs size, due date, and grade impact — flag now: Canvas doesn't reliably expose assignment weighting, so you'll likely need to tell Cap the weight once per course and it remembers. Morning/evening briefing, triggered by a scheduled local notification that opens straight into a briefing view. Apple Calendar write-back (create/edit events).

**Phase 2 — Outside accounts and location.** Google Calendar + Gmail via OAuth (Gmail's broader scopes require Google's app verification process if this ever leaves your own test devices — worth knowing up front, not a phase-2-week-3 surprise). Microsoft Graph for Outlook mail and calendar via MSAL, same shape. Email triage view — draft replies sit for your approval, nothing sends itself, ever. Location: CoreLocation + MapKit for "leave by" estimates recomputed on app open and on each opportunistic background refresh (not live-tracked every minute — that's not a real iOS capability for a third-party app); geofencing for up to 20 key places (dorm, library, gym, key buildings).

**Phase 3 — Proactive layer.** Background-refresh-driven re-scheduling of nudges and leave-by alerts. Escalating reminders for things you've dismissed repeatedly. An interruption budget: a cap on notifications per time window, with overflow batched into the next scheduled briefing instead of pinging immediately — this is logic Cap runs locally, no permission needed beyond what you already granted. Local preference/conversation memory that actually adjusts behavior (start with simple heuristics — e.g., "user always pushes back study blocks before 10am" — not on-device fine-tuning, which isn't realistically available to a third-party app).

**Phase 4 — Polish.** Lock Screen widget / Live Activity showing next leave-by and next deadline. Action Button or Shortcut bound to "start talking to Cap." Per-category cloud toggle (e.g., allow cloud for general questions, never for email drafting) so you control exactly when it phones out. Persona tuning pass based on a few weeks of actual use.

## 4. The Short, Blunt Version

Cap can be a genuinely useful, mostly-offline, privacy-tight assistant. It cannot be a literal Jarvis that's always listening and always running in the background — nothing on iOS can be that, for any developer, including Apple's own first-party apps outside of Siri's reserved hooks. The plan above gets as close as the platform allows: instant on-device brain for your own data, a one-tap voice session instead of ambient listening, and scheduled-ahead reminders instead of a process that never sleeps.

Ready to start Phase 0 when you are.
